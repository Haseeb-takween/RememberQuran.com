# Milestone 2 — Audio & Recitation: Implementation Plan

## Context

Milestone 1 shipped the reading experience. Milestone 2 adds listening: reciter
audio (2 reciters, scalable to 20+), full playback controls (per-ayah,
continuous surah, repeat for memorisation, speed), a persistent mini player,
real-time word-by-word highlight sync, click-to-hear word audio, and Quran
Radio. M1 deliberately left hooks for this: `ArabicWord` had stubbed
`isHighlighted`/`onWordClick` props, `Word.audio_url` was already fetched,
disabled Play buttons sat in `ReaderControls`/`AyahBlock`, and
`[surahId]/layout.tsx` reserved a `data-slot="audio-dock"` div.

**Hard constraints:**

1. **Zero regression on M1** — strictly additive; when no audio plays, the app
   renders exactly as before.
2. **Best- and worst-case handling** — every feature degrades gracefully
   (network failures, missing/malformed timing data, iOS restrictions).

## Verified data sources (live-verified 2026-07-15 — full detail in [audio-sources.md](audio-sources.md))

**Primary: QDC chapter-audio endpoint** (same open API family as the verses fetch):

```
GET https://api.qurancdn.com/api/qdc/audio/reciters/{recitationId}/audio_files?chapter={n}&segments=true
```

- **One MP3 per surah** + `verse_timings` (per-verse `timestamp_from/to` and
  word `segments`). Gapless continuous playback; prev/next/repeat are
  `currentTime` seeks; timings arrive in the same response.
- **Segment tuple = `[wordPosition, startMs, endMs]`**, 1-based, matching
  `Word.position` of `char_type_name === "word"` words. Timestamps are
  absolute within the chapter file.
- **Recitation IDs: Alafasy = 7, Al-Sudais = 3.** Both have word segments.
- Rejected alternative: `v4/recitations/{id}/by_chapter/{n}` — per-ayah files,
  no timings, src-swap gaps, 286 requests for Al-Baqarah.
- **Word-level audio:** `Word.audio_url` is relative (`wbw/001_001_001.mp3`)
  → prefix `https://audio.qurancdn.com/`. End markers carry `null`.

**Data-quality gotchas (all handled in `src/lib/wordSync.ts`):** malformed
segment entries exist (Alafasy 1:3 interleaves single-element arrays); Sudais
timestamps are floats; a verse's first segment can start before its
`timestamp_from` (resolve verse first, then word); gaps between segments keep
the previous word highlighted to avoid flicker.

## Architecture

**React Context + a tiny external store — no zustand** (codebase is context-based).

- **`AudioPlayerContext`** ([src/context/AudioPlayerContext.tsx](../src/context/AudioPlayerContext.tsx)) —
  owns a singleton hidden `<audio>` element via ref; `useReducer` for
  rare-change state (`status`, `mode: surah|radio`, `repeat`, `chapterId`,
  `versesCount`, effective `hasWordTiming`, error). Actions: `playVerse`,
  `playChapter`, `startRadio`, `togglePlayPause`, `nextAyah`/`prevAyah`,
  `seekToVerse`, `setReciter`, `setSpeed`, `setRepeat`, `playWord`, `retry`,
  `stop`. Loop internals live in refs — no re-renders.
- **Second, actions-only context** (same file) — referentially stable across
  state changes so high-count consumers (every `ArabicWord`) never re-render
  because of the player.
- **`playbackStore`** ([src/lib/playbackStore.ts](../src/lib/playbackStore.ts)) —
  module-level store for the ~60 Hz position `{ verseKey, wordPosition, timeMs }`,
  written by a `requestAnimationFrame` loop reading `audio.currentTime`.
  Notifies only when word/verse/second actually changes (~2 Hz playing, never
  idle). Components subscribe via `useSyncExternalStore` with per-verse
  selectors — only the recited verse re-renders.
- **Persistence:** `{ reciterId, speed }` in localStorage key
  `rq-audio-settings` with its own migration guard. `rq-reader-settings` is
  never touched.

## Failure modes & degradation

| Scenario | Detection | Behavior |
|---|---|---|
| Audio JSON fetch fails (network/404/CORS) | fetch rejects | Error state + inline retry in the bar; cache entry evicted so retry refetches; reader fully usable |
| MP3 fails mid-playback | `error` event | Error + retry near the same verse; radio auto-skips once, stops after 2 consecutive failures |
| Buffering / slow network | `waiting` event | Spinner on bar and pressed button; controls stay responsive |
| Segments missing for a reciter | checked at load | Effective `hasWordTiming: false` → plain playback with per-ayah indication (brief permits this) |
| Malformed/float/overlapping segments | `sanitizeTimings` | Dropped/normalized before use; never throws |
| Segment matches no rendered word | equality miss | Nothing highlights; previous word kept on gaps |
| `word.audio_url` null | `getWordAudioUrl` | Speaker button hidden; click is a no-op |
| iOS autoplay policy | by construction | All playback starts from a gesture; queue advances reuse the same unlocked element; `play()` rejection → paused, no crash |
| Interruption (call, tab background) | `pause` event | State syncs to paused; resume is a fresh gesture |
| Rapid actions (spam play/pause, reciter switch mid-load, navigate mid-play) | load-token + play-promise guards | Stale loads discarded; navigation never stops audio |
| Corrupt/unavailable `rq-audio-settings` | migration guard | Defaults (Alafasy, 1×); `rq-reader-settings` untouched |
| Unknown persisted reciter id | `getReciter` fallback | Default reciter, no crash |

## Non-regression guarantees

- All new props optional with defaults = previous behavior; while idle the DOM
  is identical to M1 (dock `h-0`, bar renders nothing) — verified via SSR diff.
- Touch tap-word → meaning popover unchanged; desktop hover tooltip unchanged.
- No changes to `quranApi.ts`, `types/quran.ts`, `ReaderSettingsContext`, the
  `rq-reader-settings` schema, fonts, display modes, keyboard nav, or routing.
- Highlight is a background-color change only — Arabic text is never animated
  or reflowed.
- Idle sessions produce zero player-driven re-renders (store notifies nothing).
